Action ID: marketplace/github/lock

Author: Grant Birkinbine

Publisher: github

Repository: github.com/github/lock

Description

Lock Action to support deployment locking for the branch-deploy Action

Categories

Deployment Infrastructure Communication

Categorization

Primary: Deployment

Confidence: high

Reasoning: The action is focused on managing deployment locks, which directly relates to deployment processes. Additionally, it involves communication through comments for triggering actions.

Tags: deployment lock GitHub Actions infrastructure communication

Inputs

NameRequiredDescription
github_token Yes The GitHub token used to create an authenticated client - Provided for you by default!
environment Yes The explict environment to apply locking actions to when running in headless mode - OR the default environment to use when running in the context of IssueOps commands in a comment - Examples: production, development, etc - Use "global" for the global lock environment
environment_targets No Optional (or additional) target environments to select for use with lock/unlock. Example, "production,development,staging". Example usage: `.lock development`, `.lock production`, `.unlock staging`
lock_trigger No The string to look for in comments as an IssueOps lock trigger. Used for locking branch deployments on a specific branch. Example: "lock"
unlock_trigger No The string to look for in comments as an IssueOps unlock trigger. Used for unlocking branch deployments. Example: "unlock"
reaction No If set, the specified emoji "reaction" is put on the comment to indicate that the trigger was detected. For example, "rocket" or "eyes"
lock_info_alias No An alias or shortcut to get details about the current lock (if it exists) Example: ".info"
global_lock_flag No The flag to pass into the lock command to lock all environments. Example: "--global"
prefix_only No If "false", the trigger can match anywhere in the comment
mode No The mode to use "lock", "unlock", or "check". If not provided, the default mode assumes the workflow is not headless and triggered by a comment on a pull request - Example: .lock / .unlock
reason No Reason for claiming the deployment lock for this repository

Outputs

NameDescription
triggered The string "true" if the trigger was found, otherwise the string "false"
comment_id The comment id which triggered this deployment (if it was not headless)
type The type of trigger which was found - 'lock', 'unlock', or 'info-info-alias'
comment_body The comment body which triggered this action (if it was not headless)
headless The string "true" if the run was headless, otherwise the string "false" - Headless in this context would be if the "mode" was set and the Action was not invoked by a comment on a pull request
locked If the mode is set to "check", this output will be "true" if a lock exists, otherwise "false"
lock_environment When running in headless mode and the "mode" is set to "check", this output will be the environment name that holds the lock, otherwise it will be empty
branch If the mode is set to "check", this output will be the branch name that holds the lock, otherwise it will be empty
created_by If the mode is set to "check", this output will be the user that holds the lock, otherwise it will be empty
created_at If the mode is set to "check", this output will be the ISO 8601 format date that the lock was claimed, otherwise it will be empty
reason If the mode is set to "check", this output will be the reason the deployment lock was claimed, otherwise it will be empty
link If the mode is set to "check", this output will be the link to the GitHub issue comment or action run that claimed the lock, otherwise it will be empty
global_lock_claimed The string "true" if the global lock was claimed
global_lock_released The string "true" if the global lock was released
