Author: Grant Birkinbine

Description

IssueOps commands in GitHub Actions

Categories

Communication Code quality Testing

Categorization

Primary: Communication

Confidence: high

Reasoning: This action facilitates communication through IssueOps commands in GitHub Actions, which can be related to code quality checks and testing workflows based on comments. It also uses a GitHub token for authentication, but its primary function revolves around communication and command execution.

Tags: IssueOps GitHub Actions commands automation workflow

Inputs

NameRequiredDescription
github_token Yes The GitHub token used to create an authenticated client - Provided for you by default!
status Yes The status of the GitHub Actions - For use in the post run workflow - Provided for you by default!
command Yes The string to look for in comments as an IssueOps trigger/command. Example: ".lint"
reaction Yes If set, the specified emoji "reaction" is put on the comment to indicate that the trigger was detected. For example, "rocket" or "eyes"
success_reaction Yes The reaction to add to the comment that triggered the Action if its execution was successful
failure_reaction Yes The reaction to add to the comment that triggered the Action if its execution failed
allowed_contexts Yes A comma separated list of comment contexts that are allowed to trigger this IssueOps command. Pull requests and issues are the only currently supported contexts
permissions Yes The allowed GitHub permissions an actor can have to invoke IssueOps commands - Example: "write,admin"
allow_drafts Yes Whether or not to allow this IssueOps command to be run on draft pull requests
allow_forks Yes Whether or not to allow this IssueOps command to be run on forked pull requests
skip_ci Yes Whether or not to require passing CI checks before this IssueOps command can be run
skip_reviews Yes Whether or not to require reviews before this IssueOps command can be run
param_separator Yes The separator to use for parsing parameters in comments in IssueOps commands. Parameters will are saved as outputs and can be used in subsequent steps
allowlist No A comma separated list of GitHub usernames or teams that should be allowed to use the IssueOps commands configured in this Action. If unset, then all users meeting the "permissions" requirement will be able to run commands. Example: "monalisa,octocat,my-org/my-team"
allowlist_pat No A GitHub personal access token with "read:org" scopes. This is only needed if you are using the "allowlist" option with a GitHub org team. For example: "my-org/my-team"
skip_completing Yes If set to "true", skip the process of completing the Action. This is useful if you want to customize the way this Action completes - For example, custom reactions, comments, etc
fork_review_bypass Yes If set to "true", allow forks to bypass the review requirement if the operation is being made on a pull request from a fork. This option is potentially dangerous if you are checking out code in your workflow as a result of invoking this Action. If the code you are checking out has not been reviewed, then you might open yourself up to a TOCTOU vulnerability. You should always ensure that the code you are checking out has been reviewed, and that you checkout an exact commit sha rather than a ref.
